A repository for shipping anything that can be drawn as HTML: UI prototypes,
architecture diagrams, module/structure diagrams, business flowcharts, state
diagrams, design write-ups — if it can be expressed in HTML/SVG/CSS/JS, drop it in the
right directory, push, and it's automatically built and published as a live link, no
frontend framework, no backend, no waiting on a release cycle.

**Live Portal: https://hyperbdr.github.io/product-prototypes/**

## 🧩 1. For prototype authors (read this first)

### What this is

- A "prototype" = one self-contained HTML page (with its own CSS / JS / mock data) —
  **not limited to UI screens**: architecture diagrams, module/structure diagrams,
  business flowcharts, state diagrams, design write-ups all count, as long as they're
  built in HTML.
- Drop it in the right directory, push to `main`, and within seconds it's live at a
  URL you can hand to product, engineering, or a customer for review.
- No `npm install`, no Vue/React/Vite — a finished `index.html` is the whole deliverable.

### Where your prototype goes

```text
products/<product-id>/prototypes/<your-prototype-id>/
  ├── prototype.json   # required — id / name / description / status
  ├── index.html         # required — single file, inline <style>, native JS
  └── thumbnail.svg      # recommended — thumbnail shown on the Portal card
```

Five product directories already exist: `hyperbdr` / `sourcelens` / `hyperfilelens` /
`devmind` / `hypercdr`.

Fastest path — one command scaffolds all three files:

```bash
npm run create -- --product hyperbdr --id dashboard-redesign --name "Dashboard Redesign"
```

You can also just ask an AI tool (Claude Code, Codex, etc.) to write it — point it at
the path above and have it read **[AGENTS.md](AGENTS.md)** first; the rules and the
standard workflow both live there.

### Why "dynamic" prototypes are worth the extra effort

Compared to a static mockup, a dynamic prototype can be clicked, tabbed through, and
walked end to end — reviewers don't have to imagine what happens after a click, which
cuts review round-trips a lot. Tabs, step indicators, state highlighting — all of
these "transitions" take a few lines of native JS, no extra tooling or framework
required.

### Not just UI — architecture, flowcharts and state diagrams belong here too

Most things people reach for a diagramming tool for can be built in HTML just as well,
so "prototype" here isn't limited to product screens. The same directory structure and
the same publish flow also work well for:

- **Architecture / structure diagrams**: draw nodes and edges in HTML + SVG, click a
  node to expand detail or jump to a spec — carries a lot more information than a
  static PNG, and it's trivial to update.
- **Business flowcharts / sequence diagrams**: pair tabs, step indicators, and
  progressive reveal to actually demonstrate "what happens, then what happens next,"
  instead of describing it in a paragraph.
- **State diagrams**: click a state node to highlight its transitions and triggers —
  clearer than a static state-machine image.
- **Design write-ups / tech specs**: mixed text and diagrams with collapsible
  sections — a "clickable" living document instead of a PDF you can only scroll.

The workflow is identical to a UI prototype: create a directory, write
`prototype.json` + `index.html`, push — and you have a shareable link, with no need to
buy a diagramming tool or schedule a review just to walk through an architecture
diagram.

### Two rules you must not break

> [!IMPORTANT]
> **One prototype = one self-contained directory — never entangle two of them.** Don't
> reference a file in another prototype's directory, and don't touch another
> product's or prototype's content in passing. That's what lets any single prototype
> be copied, deleted, or moved without breaking anything else. The authoritative
> isolation rules live in **[AGENTS.md](AGENTS.md)** — both humans and AI tools
> (Claude Code, Codex, etc.) follow that file, so isolation reviews stay consistent.

> [!WARNING]
> **Never hand-edit `portal/prototypes.generated.json`.** It's generated by the build
> script; any manual edit is silently overwritten on the next `npm run generate`.

### Managing versions (keeping V1 / V2 around)

If a redesign doesn't need its previous round to stay independently viewable, just
edit `index.html` in place — git history is your record.

If it does (a stakeholder wants to keep comparing V1 and V2 side by side), don't
overwrite the old directory — create a new, fully independent, versioned one:

```text
products/hyperbdr/prototypes/dashboard-redesign-v1/
products/hyperbdr/prototypes/dashboard-redesign-v2/
```

Set the old version's `status` to `archived` in its `prototype.json` (hidden from the
Portal's default view, still reachable via the "Archived" filter or its direct URL),
and keep iterating in the new directory as `draft → review → approved`. Version
history stays browsable from the Portal itself, not just `git log`.

### Before you commit

```bash
npm run validate   # checks every prototype.json / product.json
npm run generate    # rebuilds the Portal manifest
```

Both must pass before you commit — CI re-runs them and blocks publishing otherwise.

### Going live

Push to `main` and GitHub Actions builds and deploys automatically. Within seconds it's
visible at: **https://hyperbdr.github.io/product-prototypes/**

## 🖥️ 2. Local preview (optional)

You don't strictly need a local server — pushing to `main` and checking the deployed
result is usually the path of least friction. But while iterating, a local look is
faster:

```bash
npm install
npm run serve
```

Open `http://localhost:8080/` — behavior matches GitHub Pages exactly (relative paths,
`fetch()` of local JSON, all work). Single-file prototypes that don't `fetch()`
anything can still be opened by double-clicking `index.html`.

## 🛠️ 3. Maintaining the repo (Portal / scripts / CI)

The rest of this document is for people maintaining the repository itself, not just
adding prototypes.

### Repository layout

```text
index.html                 Portal shell
404.html                   Not-found page, subpath-safe
portal/                    Portal JS/CSS + generated manifest (do not hand-edit the .json)
products/<id>/product.json + prototypes/<id>/{prototype.json,index.html,...}
templates/basic/           Starter template used by `npm run create`
scripts/                   generate/validate/create/serve/build-dist, Node stdlib only
docs/                      Field reference, adding-a-prototype guide
test/                      node:test suite
.github/workflows/         CI + Pages deployment
AGENTS.md                  Repo rules AI tools and humans both follow (source of truth for isolation)
CLAUDE.md                  Pointer to AGENTS.md — see below
```

### How Portal auto-discovery works

`scripts/generate-portal.mjs` scans `products/*/product.json` and
`products/*/prototypes/*/prototype.json`, and writes
`portal/prototypes.generated.json`, which the Portal page reads to render everything.
Adding or removing a prototype never requires touching Portal code.

### Metadata fields

Full field reference, defaults, thumbnail resolution order:
[docs/metadata-reference.md](docs/metadata-reference.md). Full step-by-step guide
(including how the "updated at" timestamp is computed):
[docs/adding-a-prototype.md](docs/adding-a-prototype.md).

Prototype status values:

| Value | Portal label | Shown by default? |
|---|---|---|
| `draft` | Draft | yes |
| `review` | In Review | yes |
| `approved` | Approved | yes |
| `archived` | Archived | no — toggle the "Archived" filter in the Portal |

### Commands

| Command | Purpose |
|---|---|
| `npm run validate` | Checks every `product.json` / `prototype.json`, non-zero exit on error |
| `npm run generate` | Rebuilds `portal/prototypes.generated.json` |
| `npm run build` | `validate && generate` |
| `npm run dist` | Assembles the GitHub Pages publish artifact into `dist/` |
| `npm run serve` | Local static server at `http://localhost:8080/` |
| `npm run create -- --product <id> --id <id> --name "..."` | Scaffolds a new prototype from the template |
| `npm test` | Runs `node --test` (`test/`, built-in test runner, no extra dependency) |

### How CLAUDE.md and AGENTS.md relate

**`AGENTS.md` is this repo's authoritative rulebook** for AI tools (and humans) — hard
isolation rules, standard workflow, versioning convention. `CLAUDE.md` is just a
pointer that says "read AGENTS.md" — so the two files can never drift out of sync.

> [!NOTE]
> Rules are authoritative in `AGENTS.md`. This README's "for authors" section is a
> human-readable summary — if it ever disagrees with `AGENTS.md`, `AGENTS.md` wins;
> please open a PR to bring the README back in line.

### Enabling GitHub Pages for the first time

1. In GitHub: **Settings → Pages → Build and deployment → Source** → select
   **GitHub Actions**.
2. Push to `main` (or trigger the workflow manually from the **Actions** tab).
3. The workflow's `deploy` job prints the published URL once it finishes.

### Design constraints

- No database, no backend service, no auth — this is a static prototype gallery.
- No screenshot automation (Playwright/Puppeteer) — thumbnails are either provided by
  the prototype author or auto-generated client-side as a placeholder.
- No monorepo tooling, no bundler. Node.js is only used to scan / validate / generate /
  preview — never a runtime dependency of a prototype itself.
